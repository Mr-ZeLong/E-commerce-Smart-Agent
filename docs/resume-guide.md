# 电商售后智能助理系统 — Agent 应用开发

## 项目描述
电商售后智能助理，承接订单查询、政策咨询、退货申请等高并发咨询。针对售后政策文档更新频繁、退款规则复杂导致的人工成本高、响应慢等痛点，通过 LLM + Agent 工作流实现大部分咨询自动闭环，并建立风控机制将高风险或低置信度请求安全转交人工，避免误操作资损。

## 项目技术
LangGraph / LangChain、FastAPI、SQLModel、PostgreSQL、Qdrant、Redis、通义千问 / Qwen

## 职责描述
1. **LangGraph Agent 工作流升级**：主导 LangGraph 工作流升级至 1.0+，使用 `Command` API 显式编排 router → policy/order → evaluator → (低置信重试 → router \| 通过 → decider) 多节点链路，替代原 Supervisor 隐式路由，解决黑盒路由难以定位断点的问题，使 Agent 执行轨迹全链路可观测。

2. **RAG 检索系统优化**：针对政策文档含大量表格与层级标题、直接切分导致语义断裂的问题，基于 Qdrant 搭建向量知识库，集成 Embedding、查询重写、混合检索（稠密 + 稀疏）与精排器，降低政策问答中的事实性幻觉。

3. **置信度评估与人机回环（HITL）**：基于线上 badcase 迭代设计多维度置信度评分（意图清晰度、检索相关性、工具调用成功率、输出一致性等），构建 evaluator_node 对 Agent 输出自动打分。低置信度或大额退款等高风险场景自动触发人工审核，实现自主决策与资损防控的平衡。

4. **Agent 工具与 Prompt 工程**：针对订单查询、退货规则校验等场景设计结构化 Tool Schema 与 ReAct Prompt，约束 LLM 输出为固定 JSON 格式；配合 Pydantic 输出校验与规则引擎二次校验，拦截模型幻觉出的无效退款理由，提升工具调用成功率。

5. **意图识别与多轮澄清机制建设**：设计分层意图识别体系（一级业务域 / 二级动作 / 三级子意图），基于 Redis 会话实现槽位提取、缺失参数澄清与话题切换检测；通过 IntentRecognitionService 统一编排分类器、澄清引擎、槽位验证器与安全过滤器，提升复杂对话场景下的意图捕获准确率与对话连贯性。

6. **管理员审核决策引擎与异步任务体系**：构建 B 端管理员审核 API 与任务队列，基于金额风控阈值（¥500 / ¥2000）自动分流低/中/高风险退款申请；通过 Celery 异步执行退款支付、短信通知与管理员通知，结合 WebSocket 实时状态同步实现高并发售后场景下的资损防控与审核闭环。

7. **LLM 应用稳定性建设**：设计异常降级与兜底策略， evaluator_node 低置信场景自动回退重试，路由层设置最大迭代保护防止无限循环；通过统一 correlation_id 链路日志追踪请求全生命周期，结合关键节点日志记录工具调用结果与置信度分数，支持快速定位线上 badcase 并持续迭代优化模型表现。

# 简历写作规范

1. **去 Demo 味**：项目名称要贴近真实业务（避免 Smart/Demo/Toy 等命名），强调业务场景、用户量级、成本或效率痛点。

2. **先讲业务痛点，再讲技术方案**：项目描述开篇说明"解决了什么问题"，而非直接罗列技术特性。

3. **体现工程落地与迭代**：使用"主导升级""基于线上 badcase 迭代""从 Supervisor 迁移"等措辞，传达真实踩坑、持续优化，而非从零搭玩具。

4. **聚焦 Agent 岗位核心能力**：内容围绕 LangGraph 工作流编排、RAG 检索与幻觉治理、置信度评估与人机回环（HITL）、Prompt / Tool 工程、可观测性与稳定性建设展开。

5. **职责描述按「方法 → 解决的问题 → 结果/价值」组织**：每条以动词开头，给出具体技术方法，点明业务痛点，最后量化或定性价值。

6. **技术栈精简、控制字数**：技术栈用顿号分隔，控制在 1 行；全文聚焦后端与 Agent 逻辑，不写前端、异步通信等边缘内容，总字数控制在 600 字以内。

7. **注入业务风险意识**：强调真实系统才会关注的概念，如"资损防控""误操作""高风险场景""兜底策略""badcase 迭代"等，增强真实感与可信度。
