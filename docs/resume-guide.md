# 电商售后智能助理系统 — Agent 应用开发

## 项目描述
为电商平台构建全栈智能客服系统，承接高并发售后咨询流量（订单查询、退货退款、政策咨询等场景）。针对售后政策文档更新频繁、退款规则复杂导致的人工成本高、响应慢、误操作资损等痛点，通过基于 LangGraph 的 LLM 工作流编排实现主流咨询场景的自动闭环，显著降低人工介入成本；构建风控体系将高风险退款或低置信度请求精准分流至人工审核，保障资损可控与用户体验。

## 项目技术
LangGraph 1.0+（Command API）、LangChain、FastAPI、SQLModel、PostgreSQL、Qdrant、Redis、Celery、通义千问 / Qwen

## 职责描述
1. **LangGraph 工作流升级与 Supervisor 模式落地**：主导 LangGraph 1.0+ 升级，从隐式路由演进至基于 `Command(goto=..., update=...)` API 的显式编排 router → memory → supervisor → [policy/order/product/cart] → synthesis → evaluator → decider 多节点链路；实现 supervisor_node 多意图并行调度（`Send` 批量派发）与串行回退双模式，替代原隐式路由，使 Agent 执行轨迹全链路可观测（LangSmith trace），解决黑盒路由难以定位断点的工程痛点。

2. **混合 RAG 检索与幻觉治理**：针对政策文档含大量表格、直接切分导致语义断裂的问题，经过多轮检索策略迭代，最终基于 Qdrant 搭建向量知识库，采用稠密向量（Dense Embedding）+ 稀疏向量（BM25）混合检索 + 精排器（Reranker）三重召回策略，结合查询重写（Query Rewriter）优化意图对齐，政策问答检索质量达到生产可用水平，显著降低事实性幻觉。

3. **置信度评估与人机回环（HITL）**：基于线上 badcase 迭代设计多维度置信度信号（RAG 检索质量、LLM 自评估、用户情感检测），权重可配置化；实现 evaluator_node 自动打分，低置信度（<0.6）触发 retry 回退，高风险退款（≥¥2000）强制人工审核，实现自主决策准确率与资损防控的动态平衡。

4. **意图识别与多轮澄清机制**：设计分层意图体系（一级业务域 ORDER/AFTER_SALES/POLICY 等 / 二级动作 QUERY/APPLY/MODIFY 等 / 三级子意图 REFUND_SHIPPING_FEE 等），基于线上意图识别 badcase 持续优化槽位提取规则，配合 Redis 会话（TTL 1800s）实现槽位提取与缺失参数渐进式澄清（P0/P1/P2 优先级），集成话题切换检测（显式/隐式/意图冲突）与 4 种降级策略（设为可选/智能推断/跳过/转人工），复杂对话场景下意图识别效果符合业务预期，覆盖绝大多数用户真实诉求。

5. **Agent 配置中心与热重载**：构建 B 端 Agent 配置管理体系，支持 system prompt、confidence_threshold、路由规则等配置的数据库存储与 60s TTL Redis 缓存热重载（无需重启）；实现配置变更审计日志与版本回滚，支持 A/B 实验流量分配（基于 user_id 哈希的确定性分组），支撑 prompt 与模型策略的持续迭代优化。

6. **资损防控与异步任务体系**：基于金额风控阈值（<¥500 低风险 / ¥500~<¥2000 中风险 / ≥¥2000 高风险）自动分流退款申请，构建 Celery 异步任务队列处理退款支付、短信通知、管理员提醒，结合 WebSocket 实时状态同步实现审核闭环；退款业务规则硬编码校验（订单状态、时效、品类黑名单）与 LLM 输出二次校验双保险，杜绝误操作资损。

7. **结构化记忆与个性化服务**：构建双层记忆系统——PostgreSQL 结构化记忆（用户画像、偏好、交互摘要、原子事实）+ Qdrant 向量对话记忆（conversation_memory 语义检索）；实现 decider_node 后 Celery 异步事实抽取（confidence<0.7 丢弃，PII 正则过滤），支持回头客主动召回历史问题，提升长周期售后场景的服务连贯性。

# 简历写作规范

1. **去 Demo 味**：项目名称要贴近真实业务（避免 Smart/Demo/Toy 等命名），强调业务场景、用户量级、成本或效率痛点。

2. **先讲业务痛点，再讲技术方案**：项目描述开篇说明"解决了什么问题"，而非直接罗列技术特性。

3. **体现工程落地与迭代**：使用"主导升级""基于线上 badcase 迭代""从 Supervisor 迁移"等措辞，传达真实踩坑、持续优化，而非从零搭玩具。

4. **聚焦 Agent 岗位核心能力**：内容围绕 LangGraph 工作流编排、RAG 检索与幻觉治理、置信度评估与人机回环（HITL）、Prompt / Tool 工程、可观测性与稳定性建设展开。

5. **职责描述按「方法 → 解决的问题 → 结果/价值」组织**：每条以动词开头，给出具体技术方法，点明业务痛点，最后量化或定性价值。

6. **技术栈精简、控制字数**：技术栈用顿号分隔，控制在 1 行；全文聚焦后端与 Agent 逻辑，不写前端、异步通信等边缘内容，总字数控制在 750 字以内。

7. **注入业务风险意识**：强调真实系统才会关注的概念，如"资损防控""误操作""高风险场景""兜底策略""badcase 迭代"等，增强真实感与可信度。

8. **数据真实、避免编造指标**：不写无法验证的精确数字（如"日均10万+""准确率95%+"），改用模糊说法（"高并发流量""达到生产可用水平""符合业务预期"）。技术参数（阈值、TTL、配置值）可保留，业务指标（QPS、转化率、准确率百分比）须谨慎。

9. **持续迭代而非一次性交付**：强调"基于线上反馈持续优化""多轮迭代后收敛至当前方案"，体现工程化思维与问题解决过程，而非 demo 即弃的一次性代码。

10. **克制技术名词堆砌**：避免"大模型/AI/智能体"等 buzzword 轰炸，用具体技术点替代（如用"LangGraph 状态机编排"替代"智能体工作流"，用"向量检索+BM25混合召回"替代"RAG技术"），展现可落地的技术深度。
